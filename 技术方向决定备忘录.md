# 技术方向决定备忘录

> **用途**：本备忘录是本软件的**技术方向锚点**，目的是在后续交给 Claude Code / Codex 实现时，避免每次都要重新讨论技术选型。本文不是完整的技术设计文档，而是关键决定的清单，等真正动手实现时，工程方需要在此基础上展开为完整的架构设计。

---

## 一、产品定位回顾

跨平台（macOS + Windows）桌面工作台软件，面向短视频带货运营者，聚合"任务单生成 + AI 内容生产 + 数据库管理"等多个工作流。macOS 与 Windows 两端功能完全一致，用户可以在任一端完成创作、管理、任务单导出、状态查看和设置维护。**本地优先 + 云同步友好**：所有数据在本地，但通过 OneDrive / 坚果云这类工具支持多台工作电脑切换；影刀 RPA 作为外部 Windows 执行器读取任务单并回写状态。

---

## 二、技术栈决定

| 层          | 选择                             | 一句话理由                                 |
| ---------- | ------------------------------ | ------------------------------------- |
| 桌面壳        | **Electron**                   | 生态最成熟，Claude Code 训练数据最丰富，AI 写代码出错率最低 |
| 前端框架       | **React + TypeScript**         | shadcn 生态兼容性最好，AI 友好度最高               |
| UI 组件      | **shadcn/ui + Tailwind CSS**   | 可定制度高，原型代码可直接迁移到生产                    |
| 状态管理       | **Zustand**                    | 轻量，避免 Redux 样板代码                      |
| 本地数据库      | **SQLite + better-sqlite3**    | 标准答案，10 万行无压力，同步 API 简化逻辑             |
| 文件监听       | **chokidar**                   | 跨平台稳定，OneDrive/坚果云同步友好                |
| 视频处理       | **ffmpeg 二进制 + fluent-ffmpeg** | 拉帧、缩略图序列生成的标准方案                       |
| 表格读写       | **exceljs**                    | 比 xlsx 库支持的格式更完整                      |
| LLM 调用     | **官方 SDK + 自建 provider 抽象层**   | 多家模型必须可切换                             |
| 图片 API     | **自建 provider 抽象层**            | GPT Image / 即梦 / 可灵等可扩展               |
| GitHub API | **Octokit**                    | 用于云端 Skill 仓库同步                       |
| 打包分发       | **electron-builder**           | 双平台签名标准方案                             |

### 不选的方案及理由

- **Tauri**：更轻、更快，但 Rust 后端对 AI 写代码不友好，本产品不是消费级海量分发，省的代价不在瓶颈上。
- **Flutter Desktop**：Dart 语言对 Claude Code 不友好，桌面生态不够成熟。
- **Web App**：本地文件密集型，无法满足。

## 三、关键架构原则

### 3.1 进程分工铁律

- **主进程**：窗口管理、文件系统访问、SQLite 读写、ffmpeg / 子进程调用、HTTP 请求（API 调用）
- **渲染进程（前端 React）**：纯 UI、用户交互、状态管理。**严禁**直接访问文件系统或数据库
- **Worker / 子进程**：拉帧、批量文件操作、批量数据导入导出等 CPU/IO 密集活儿，避免阻塞主进程

### 3.2 IPC 契约必须类型化

- 主进程与渲染进程之间所有通信全部 TypeScript 类型化
- 推荐使用 `electron-trpc` 或自建带 `zod` 校验的 IPC 层
- **严禁** `ipcRenderer.send('do-something', anyData)` 这种无类型调用

### 3.3 性能保护

- 任何 >100 行的列表 / 网格强制使用虚拟列表（`react-window` 或 `@tanstack/virtual`）
- 视频预览**不做实时 scrubbing**，使用 ffmpeg 预拉帧的缩略图序列代替（专业剪辑软件标准做法）
- 数据库查询走索引；批量操作走事务（transaction）
- 大文件操作放在子进程里跑

### 3.4 安全

- 关闭 `nodeIntegration`
- 启用 `contextIsolation`
- 使用 `contextBridge` 暴露受限 API
- 启用 ASAR 打包

### 3.5 跨平台路径处理（关键）

- **数据库只存相对路径**（如 `videos/SKU001/style_a.mp4`）
- 工作空间配置 Mac 根路径 + Windows 根路径
- UI 展示当前设备路径；任务单导出展示并写入目标执行平台路径
- v1 默认目标执行平台为 Windows 影刀 RPA，因此 RPA 字段默认按 Windows 根路径拼接
- 导出前做"路径校验"：当前设备检查文件存在、可读、可解析；同时生成 `preflight.json` 供 Windows 端软件或 RPA 在执行前二次校验
- 云盘占位符检测不能只靠文件大小，应结合可读性探测、媒体元数据读取和目标端预检

### 3.6 云同步友好

- **任务单**是 Excel/CSV/JSON 文件（软件 → RPA 单向写）
- **状态回写**是 `<任务ID>__attempt-<序号>.json` 单条小文件，写入同目录的 `status/` 子文件夹（RPA → 软件单向写）
- 状态文件先写 `.tmp`，完成后原子重命名为 `.json`，软件只读取 `.json`
- 重试不删除旧状态文件，而是创建新的 attempt 和 run_key
- 双方只写自己的文件，永不同时写同一文件，避免 OneDrive / 坚果云同步冲突
- 同一工作空间同一时间只允许一个 App 实例写入 `workspace.db`；另一台电脑打开同一空间时必须通过锁文件和心跳检测进入只读或拒绝打开，避免云盘同步 SQLite 冲突

### 3.7 Skill 系统的 include 机制

- `{{include: xxx.md}}` 是**软件层面的字符串替换**，不是让 LLM 自己读文件
- 调用 LLM 之前，软件先把所有 include 标记替换为对应文件实际内容，拼成一段完整文本
- 等价于模板引擎（Jinja2 / Handlebars 思路）
- LLM 收到的就是一段普通的长 system prompt，与具体模型无关
- include 解析必须限制在当前 Skill 根目录内，禁止 `../`、绝对路径、符号链接越界、隐藏目录、`.snapshots/` 和非 Markdown 文件；任何解析错误都必须中止 LLM 调用

### 3.8 凭证与跨设备同步

- 默认凭证存储使用系统能力：macOS 钥匙串 / Electron `safeStorage`，Windows DPAPI / Credential Manager / Electron `safeStorage`
- 默认 API key 加密结果绑定当前系统用户和设备，不承诺跨电脑直接解密
- 若用户需要 API key 随工作空间一起同步，提供可选的跨设备凭证保险箱：使用用户主密码或恢复密钥加密后放入云盘同步目录
- 工作空间数据、业务数据库、用户自建 Skill、启用配置和任务单可随云盘同步；缓存、缩略图、临时文件和普通日志默认本机保存

## 四、反模式（必须避免）

1. ❌ 在主进程同步执行 ffmpeg 或大型数据库查询
2. ❌ 渲染进程直接访问文件系统或数据库
3. ❌ IPC 传递无类型的 `any` 数据
4. ❌ 列表 / 网格不做虚拟化
5. ❌ 视频用 `<video>` 标签实时 scrubbing 时间轴
6. ❌ 多个组件并发写同一个 Excel 文件
7. ❌ 把视频原文件复制进数据库或软件目录（只存路径）
8. ❌ 在客户端内置私有 GitHub token 或依赖“隐藏仓库 URL”保护官方 Skill
9. ❌ 两台电脑同时写同一个云盘工作空间的 `workspace.db`

## 五、推荐项目结构

```
my-workbench/
├── apps/
│   └── desktop/                   # Electron 应用
│       ├── main/                  # 主进程代码（window、IPC server、tray）
│       ├── preload/               # contextBridge 桥接脚本
│       └── renderer/              # React 前端（UI）
│           ├── components/        # 可复用 UI 组件
│           ├── pages/             # 页面（任务单、工作区...）
│           ├── stores/            # Zustand stores
│           └── lib/               # 工具函数
├── packages/
│   ├── shared-types/              # 端到端共享 TypeScript 类型
│   ├── db/                        # SQLite schema + 查询封装
│   ├── llm-providers/             # 多家 LLM 抽象层（OpenAI/Anthropic/Google/...）
│   ├── image-providers/           # 多家图片模型抽象层（GPT Image/即梦/可灵/...）
│   ├── skill-engine/              # Skill 解析 + include 展开 + 版本管理
│   ├── rpa-bridge/                # 任务单导出 + 状态文件读取
│   └── ffmpeg-utils/              # 视频拉帧、缩略图生成

│   └── ffmpeg/                    # ffmpeg 二进制（按平台分发）
└── docs/
    ├── 技术方向决定备忘录.md       # 本文件
    └── UIUX需求文档.md             # 设计文档
```

## 六、数据持久化布局

### 6.1 软件全局数据（用户隔离）

存放在系统标准目录：

- macOS: `~/Library/Application Support/MyWorkbench/`
- Windows: `%APPDATA%\MyWorkbench`

包括：

- `config.db`：全局配置（API key、用户偏好、工作空间索引）
- `skills/`：全局 Skill 池
  - `official/`：从云端拉取的官方原版（只读）
  - `user/`：用户自建或副本
- `cache/`：缩略图缓存、API 响应缓存等
- `logs/`：日志文件

### 6.2 工作空间数据（每个品牌一份，建议放 OneDrive 同步盘）

```
品牌A/
├── workspace.db              # 该空间的 SQLite 数据库（视频/标签/标题/任务等）
├── workspace.json            # 空间元数据（路径映射、平台账号等）
├── videos/                   # 视频原文件（按 SKU 子文件夹）
├── covers/                   # 封面文件（镜像 videos 结构）
├── images/                   # AI 生成的图片
│   ├── main/                 # 主图素材
│   ├── detail/               # 详情页素材
│   └── _trash/               # 软删图（按配置定期真删）
├── tasks/                    # 任务单
│   └── 2026-05-07/
│       ├── tasks.xlsx
│       ├── tasks.csv
│       ├── tasks.json
│       ├── preflight.json    # 目标端路径预检清单
│       └── status/           # RPA 写回的状态文件
│           ├── T20260507-001__attempt-001.json
│           └── T20260507-002__attempt-001.json
├── skills_config/            # 这个空间启用了哪些 Skill 的配置
└── _backup/                  # 自动备份
```

`workspace.db` 可以随工作空间目录同步，但它不是协作数据库。v1 的同步模型是“多设备切换使用”，不是“多设备同时编辑”。软件必须在工作空间根目录写入锁文件和心跳，防止同一空间被两个 App 实例同时写入。

## 七、最关键的几个第三方依赖

| 依赖                                         | 用途         | 注意事项                                  |
| ------------------------------------------ | ---------- | ------------------------------------- |
| `electron`                                 | 桌面壳        | 锁定主版本，注意安全更新                          |
| `better-sqlite3`                           | SQLite     | native 模块，需要按平台编译，electron-builder 配合 |
| `chokidar`                                 | 文件监听       | OneDrive 同步可能产生大量事件，需要 debounce       |
| `fluent-ffmpeg`                            | 视频处理       | 需要随包带 ffmpeg 二进制（每个平台对应）              |
| `exceljs`                                  | Excel      | 比 xlsx 更稳，支持样式                        |
| `octokit`                                  | GitHub API | 用于云端 Skill 同步                         |
| `zustand`                                  | 状态管理       | 简单优于 Redux                            |
| `react-window` 或 `@tanstack/react-virtual` | 虚拟列表       | 必须使用，不可省                              |
| `electron-builder`                         | 打包         | 双平台签名 + 自动更新                          |
