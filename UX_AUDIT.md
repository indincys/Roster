# Roster UX 审计报告

审计日期：2026-05-26  
审计对象：短视频运营工作台桌面端  
审计方式：阅读 `UIUX需求文档.md`、核心 React 页面源码、桌面应用真实运行截图  
验证环境：Node 22.22.3、npm 10.9.8、Electron 构建产物、临时用户数据和临时工作空间  

## 结论摘要

当前产品的主要问题不是单纯“视觉偏土”，而是大量页面仍停留在“功能被摆出来”的阶段。页面没有稳定地围绕用户当下的任务、判断对象和下一步动作来组织信息。

最典型的问题有四类：

1. 重要对象没有成为视觉中心，比如图片验收页里用户要判断图片质量，但图片只是小卡片。
2. 页面按数据库字段或功能模块平铺，而不是按真实工作流组织，比如生成、验收、入库、返工之间缺少连续路径。
3. 数据状态和操作反馈不够面向决策，比如首页和任务单有多个计数卡，但计数不一定对应用户当天最该处理的事项。
4. 不同页面的交互模型不统一，比如图片、视频、标题、文案、任务单都需要“队列 + 主对象 + 详情/动作”，但现在每个页面都重新发明一套布局。

建议不要先做全局换皮。正确顺序是先修核心工作流的信息架构，再统一视觉语言。否则界面会更好看，但仍然不好用。

## 对标产品

本项目是桌面生产力工具，参考对象应是成熟的工作台和内容处理工具，而不是营销网站。

Linear：待处理队列、状态、筛选、批量操作和键盘流转很清晰，适合参考任务单、定时任务、工作流状态。

Figma：画布是主对象，右侧属性面板服务于当前选中对象，适合参考图片、视频、封面、素材详情类页面。

Lightroom / Apple Photos：图片查看、放大、前后切换、标记、筛选是核心体验，适合参考图片工作室和图片库。

Buffer / Later：内容排期、平台账号、日历和发布状态有成熟模式，适合参考任务单和定时任务。

## 总体设计方向

视觉方向：专业运营控制台，低装饰、高密度、强对象预览、强状态可见。

布局原则：每个页面先确定主对象，再确定队列、详情和操作区。不要用统计卡和通用卡片替代工作流。

交互原则：高频动作必须靠近被操作对象，所有生成结果必须可预览、可放大、可比较、可返工、可批量处理。

## 审计方法

每个页面按以下问题审计：

1. 用户来到这里要完成什么？
2. 页面最重要的操作对象是什么？
3. 用户做决定前必须看到什么？
4. 下一步动作是否明显且贴近对象？
5. 空状态、错误状态、批量状态、回退状态是否自然？

问题标签：

- IA：信息架构问题
- Flow：工作流断裂
- Review：验收能力不足
- Batch：批量处理不足
- State：状态反馈不足
- Visual：视觉层级不足
- Density：空间密度不合理
- Copy：文案不解释当前决策

优先级：

- P0：阻碍核心任务判断或导致用户无法高效完成主流程
- P1：明显降低效率或造成理解成本
- P2：视觉一致性、低频页面体验和长期可维护性问题

## 全局问题

### G1. 页面普遍缺少“主对象”

优先级：P0  
标签：IA、Visual、Flow

当前很多页面以标题、说明、指标卡、配置卡、表格开场，但用户真正要处理的对象没有被放到视觉中心。

例子：

- 图片工作室：图片是主对象，但验收阶段图片尺寸太小。
- 任务单：任务行是主对象，但页面上半屏被配置和指标占据。
- 标题工作区：结果是主对象，但结果区空状态占据大面积，且没有预设“生成后如何筛选和入库”的工作区结构。
- 视频库：视频是主对象，但缩略图极小，右侧详情初始为空。

改造原则：

所有高频页面改成三段式：

```text
左侧：队列、筛选、批量选择
中间：主对象预览或主要表格
右侧：详情、来源、参数、操作
```

不适合三段式的页面，至少也要明确：

```text
页面主对象是什么
当前选中对象是什么
下一步动作是什么
```

### G2. 统计卡过多，但不够可操作

优先级：P1  
标签：State、Visual、Density

首页、任务单、视频库、图片工作室、数据库页都大量使用统计卡。问题是这些数字多数只是“看起来像仪表盘”，没有直接服务下一步操作。

建议：

- 首页只保留“今天需要处理的事项”和“阻塞项”。
- 高频工作页不要默认占四个统计卡，改为紧凑状态条。
- 指标必须能点击进入对应筛选结果，否则降低视觉权重。

### G3. 顶部全局状态与页面状态不一致

优先级：P1  
标签：State、Flow

顶部有“3 个待执行”“更新失败”“任务中心”等状态，但用户不容易知道它们来自哪里、是否与当前页面相关。

建议：

- 顶部状态只保留全局级事件：后台任务、更新、通知、搜索。
- 页面内状态必须在当前页面局部展示，比如图片生成待验收应在图片工作室和任务中心同步。
- 全局任务中心需要能按类型筛选：图片生成、任务单、定时任务、导出、扫描。

### G4. 数据库页和工作区页边界不清

优先级：P0  
标签：IA、Flow

产品里有“工作区”和“数据库”两类入口，但当前页面之间的跳转关系不够强。例如提示词库和图片工作室、视频库和封面工作区、标题库和任务单之间，本应有明确的“用这些去生成”或“回到来源”的路径。

建议：

- 数据库页负责管理和筛选对象。
- 工作区页负责生产、验收、返工和入库。
- 数据库选中对象后应出现上下文动作，比如“用选中提示词生成图”“给选中视频做封面”“用选中标题替换任务单标题”。

## 页面级审计

### 1. 首页 / Dashboard

当前用户任务：快速知道今天该做什么、哪里失败、下一步从哪里开始。  
当前问题优先级：P1

问题：

- 首页显示四个指标卡，但它们多数为 0 时没有告诉用户真实启动路径。
- “今日任务队列”与任务单页数据可能脱节，截图中已种子生成数据后首页仍呈现空队列感。
- “路径映射”占据很大视觉面积，但这是低频维护信息，不应压过当天任务。
- 快捷操作是按钮集合，不是按工作流排序。

建议改法：

- 首页改为“今日运营面板”，第一屏只展示三件事：今天待处理、阻塞告警、最近产物。
- 路径映射移动到设置或折叠为告警，仅在路径不可用时展示。
- 快捷操作改为工作流启动器：生成任务单、验收图片、处理封面、导出任务。

验收标准：

- 用户 5 秒内能知道今天最该点哪里。
- 所有首页数字都能点击进入对应筛选结果。
- 空状态时给出具体启动步骤，而不是只显示空。

### 2. 任务单

当前用户任务：生成今天的发布计划，检查任务行，导出给 RPA，跟踪执行状态。  
当前问题优先级：P0

问题：

- 配置区占用过高，生成后没有自动折叠成摘要。
- 当前任务表格才是核心，但初屏被指标和配置挤压。
- 任务单日期和生成数据容易出现理解断点，用户看到“0 行”时不知道是数据缺失、日期不对还是生成失败。
- 导出、扫描状态、定时、重新生成并列出现，但没有按“生成前、生成后、执行后”分阶段。
- 表格横向滚动明显，列权重没有按人工检查优先级设计。

建议改法：

任务单重构为三状态页面：

```text
未生成：配置向导 + 数据预检
已生成：任务表格为主，配置折叠成摘要
已导出/执行中：状态扫描、失败重试、导出路径预检为主
```

表格列优先级：

```text
时间、平台、账号、视频、标题、状态、错误
```

SKU、Run Key、路径等信息进入行详情或右侧详情面板。

验收标准：

- 生成前明确提示视频库、标题库、标签库、账号是否满足生成条件。
- 生成后首屏至少 60% 高度给任务表格。
- 失败行能直接重试，错误原因不被截断。
- 导出前展示 Windows RPA 路径预检结果。

### 3. 标题工作区

当前用户任务：用 Skill 和模型生成标题，比较结果，挑选入库。  
当前问题优先级：P1

问题：

- 页面结构比较接近需求文档，但生成前的大面积空白没有价值。
- 模型选择是一排标签，缺少“启用模型、失败模型、结果列”的关系。
- 入库按钮远离未来结果区，生成后应变为 sticky 批量操作条。
- 缺少标题质量评估辅助，比如字数、重复度、禁用词、平台适配。

建议改法：

- 生成前结果区展示最近生成、标题库高分样例、当前 Skill 摘要，而不是空白。
- 生成后按模型列展示，并提供列级重试、列级复制、列级入库。
- 每条标题展示质量辅助：长度、疑似重复、是否含违禁词、来源模型。

验收标准：

- 生成前页面不出现大面积无意义空白。
- 生成结果可以按模型比较。
- 勾选后出现底部 sticky 批量条。

### 4. 图片工作室

当前用户任务：生成图片，查看细节，决定通过、拒绝或重生。  
当前问题优先级：P0

问题：

- 图片验收阶段没有放大查看能力，无法完成质量判断。
- 图片卡片太小，且被统计卡、说明文字和大面积空白抢走注意力。
- 通过、拒绝动作只存在于缩略卡片下方，不支持键盘流转、连续验收、对比查看。
- “可用图片”“待验收”“结果处理”是状态，但没有形成工作队列。
- 文生图、图生图、单次、批量是功能维度，不是用户最终验收维度。

建议改法：

图片工作室第一阶段应优先改成“验收工作台”：

```text
左侧：待审图片队列，支持筛选和批量选择
中间：大图查看器，支持适配窗口、原图尺寸、缩放、拖拽、前后切换
右侧：来源提示词、模型、比例、文件路径、重生、通过、拒绝、入库状态
底部：当前批次进度和快捷键提示
```

必须新增图片查看器：

- 点击图片打开大图。
- 支持 25% 到 400% 缩放。
- 支持 Fit、100%、拖拽平移。
- 支持左右切换。
- 支持快捷键：A 通过、X 拒绝、R 重生、Space 适配窗口。
- 大图中可查看来源提示词和参数。

验收标准：

- 用户不用离开验收页就能完成连续审图。
- 图片必须成为首屏最大视觉对象。
- 拒绝后能看到软删位置，且可撤销。
- 一条提示词的多张图能横向对比。

### 5. 封面工作区

当前用户任务：为视频定帧、裁剪、保存封面，然后处理下一个视频。  
当前问题优先级：P1

优点：

- 这是目前最接近“对象工作台”的页面，有左侧队列、中间大预览、底部时间轴和快捷键。

问题：

- 视频列表只显示文件名，缺少缩略图和状态优先级。
- 预览失败时仍展示大面积黑色画布，原因和可补救动作不够明确。
- 批量首帧、应用此封面、比例选择挤在同一行，层级不够清楚。
- 时间轴缩略图在底部存在，但当前帧、关键帧、推荐帧没有辅助判断。

建议改法：

- 左侧队列显示缩略图、SKU、是否已有封面、异常状态。
- 预览失败时提供“打开原视频路径”“重新取帧”“用首帧占位”的动作。
- 保存成功后右上角 toast 不够，应在左侧队列中立即标记已完成并跳到下一个。

验收标准：

- 连续处理 20 个视频时，用户不需要鼠标离开主预览区太多。
- 失败视频能被跳过、重试、标记。
- 队列中已处理和未处理状态明显。

### 6. 文案工作区

当前用户任务：生成口播或剪映文案，复制、筛选、入库。  
当前问题优先级：P1

问题：

- 与标题工作区几乎同构，但文案是长文本，不能用标题式卡片处理。
- 缺少结构化预览，比如开场、卖点、口播、转化、时长估计。
- 复制按钮在单条结果里，但批量复制和平台适配需要更强。

建议改法：

- 文案结果用“文稿编辑器”而不是普通卡片。
- 右侧展示结构检查：字数、预计时长、禁用词、SKU 关联、平台适配。
- 提供“复制纯文本”“导出 TXT”“入库并标记用途”。

验收标准：

- 长文案在一屏内可读，可编辑，可复制。
- 生成后能快速判断是否适合 15 秒、30 秒或 60 秒视频。

### 7. 视频库

当前用户任务：扫描视频，确认素材是否可用，补充 SKU/款式，进入封面或任务单。  
当前问题优先级：P0

问题：

- 视频缩略图极小，无法判断素材内容。
- 右侧详情初始为空，首屏有大量未使用空间。
- “元数据异常”被展示为红色状态，但没有解释原因和修复路径。
- 批量操作只有选中后出现，发现性弱。
- 表格列多，但人工最关心的“内容画面、SKU、是否可发布、是否有封面”没有被突出。

建议改法：

- 视频库默认采用表格 + 详情面板，但详情面板应自动选中第一条或显示扫描诊断。
- 缩略图尺寸提高，hover 可预览短视频片段。
- 异常状态增加原因：读取失败、ffmpeg 缺失、文件未同步、格式不支持。
- 增加上下文动作：做封面、加入任务单候选、打开文件位置、重新读取元数据。

验收标准：

- 用户能直接看出哪些视频可用、哪些缺封面、哪些异常。
- 选中一条视频后，右侧立即显示可操作详情。
- 异常视频有明确处理动作。

### 8. 标签库、标题库、提示词库、文案库、图片库

当前用户任务：管理可复用内容资产，并把资产送回工作流。  
当前问题优先级：P1

共性问题：

- 数据库页是通用 CRUD 模板，缺少各资产类型自己的判断方式。
- 统计卡重复，表格列平均，右侧编辑面板容易变成字段表单。
- 筛选按钮只是按钮，缺少真实筛选模型。
- 数据库和工作区的来回路径弱。

分页面建议：

标签库：

- 按 SKU 聚合，不应只是字段表。
- 默认标签和测试标签要有比例、覆盖率、缺失 SKU 提醒。

标题库：

- 突出分数、使用次数、最近使用、重复度。
- 支持“送到任务单换标题”。

提示词库：

- 这是图片质量的上游，应突出生成数、通过率、反面标记、样例图片。
- 支持选中提示词直接进入图片生成。

文案库：

- 长文本应可展开阅读和复制，不适合只显示摘要行。
- 支持按 SKU、平台、用途筛选。

图片库：

- 必须是图片优先的图库，不应是表格优先。
- 点击图片进入大图查看器。
- 支持按场景、比例、提示词、模型、状态筛选。

验收标准：

- 每个库都有资产类型专属的主视图。
- 所有数据库对象都有“送回工作区”的操作。
- 图片库和视频库必须支持可用的视觉预览。

### 9. Skill 中心

当前用户任务：管理 Skill、编辑提示词文件、启用到当前工作空间、运行测试。  
当前问题优先级：P1

优点：

- 已有左侧列表、中间编辑、右侧配置的结构。

问题：

- 编辑区像原始代码编辑器，但 Skill 的真实用户可能更需要结构化编辑和测试反馈。
- 启用状态在左侧小图标和右侧按钮中重复，但没有告诉用户“这个 Skill 被哪些工作区使用”。
- 即时测试在右下角，离编辑主内容较远。
- 保存、保存当前文件、启用、测试、快照的关系比较复杂。

建议改法：

- 顶部增加 Skill 状态条：类型、来源、启用工作区、最近测试、最近保存。
- 编辑器旁边保留测试面板，但测试结果要能固定、对比、复制。
- 文件树和 include 标记合并成“上下文文件”区域。

验收标准：

- 用户能明确知道当前编辑的是哪个文件、是否已保存、是否已启用。
- 运行测试后能看到完整输入摘要、输出结果和错误。

### 10. 定时任务总览

当前用户任务：查看哪些自动任务会运行、下一次何时运行、失败原因是什么。  
当前问题优先级：P1

问题：

- 表格字段完整，但不够突出“下一次执行”和“最近失败”。
- 新建入口只有任务单定时，其他类型需要去工作区创建，发现性一般。
- 历史区域在底部，选中任务后反馈不够强。

建议改法：

- 左侧任务列表，右侧运行历史和配置详情。
- 失败任务置顶，并显示错误摘要。
- 新建入口应弹出类型选择，而不是只创建任务单定时。

验收标准：

- 用户能一眼看到下一次要跑什么。
- 失败任务能直接进入对应工作区修复。

### 11. 设置

当前用户任务：配置工作空间、路径、API、平台账号、备份和维护。  
当前问题优先级：P1

问题：

- 所有设置纵向堆叠，API 设置占据首屏巨大空间。
- 工作空间、API、平台账号、备份、系统策略、更新、反馈包混在一个滚动页。
- 保存反馈散落在页面中，用户不容易知道哪个配置已保存。

建议改法：

设置改成左侧分组：

```text
工作空间
AI Provider
平台账号
路径与同步
备份与恢复
更新与维护
反馈日志
```

每组右侧只处理一个任务。API Provider 应独立成表格 + 详情编辑面板。

验收标准：

- 用户能在 3 秒内找到“配置图片 API”和“配置 Windows 路径”。
- 每个保存动作都有局部状态，不依赖顶部全局消息。
- 危险操作有明确确认。

### 12. 反馈日志

当前用户任务：排查问题、打包反馈、查看最近错误。  
当前问题优先级：P2

问题：

- 导航中有反馈日志，但实际能力主要在设置页“反馈包”。
- 反馈日志应是问题排查入口，不应只是设置项。

建议改法：

- 独立页面展示最近错误、API 调用失败、定时任务失败、导出失败、缓存清理、反馈包生成。
- 支持一键打包当前问题上下文。

## 第一阶段重构建议

不要一次重做所有页面。建议按“最影响用户判断”的顺序做。

### Phase 1：修核心判断流程

优先级：最高

1. 图片工作室验收重构，新增大图查看器和连续审图。
2. 视频库重构，提高缩略图和异常诊断能力。
3. 任务单重构，生成后折叠配置，任务表格成为主体。

交付标准：

- 图片生成后可以大图验收。
- 视频库能看清素材并处理异常。
- 任务单生成后用户能直接检查和导出。

### Phase 2：统一资产库和工作区的关系

优先级：高

1. 提示词库到图片生成的选中流转。
2. 图片库改为图库优先。
3. 标题库、文案库加入“送回任务单/工作区”的上下文动作。

交付标准：

- 数据库不只是 CRUD，而是生产工作流的一部分。
- 每个资产都有来源和去向。

### Phase 3：统一生成类页面

优先级：中

1. 标题工作区和文案工作区共用生成结果模式。
2. 多模型结果列统一。
3. 生成中、失败、重试、入库、复制的状态统一。

交付标准：

- 用户学会一个生成页，就能理解其他生成页。

### Phase 4：低频系统页整理

优先级：中低

1. 设置页分组。
2. Skill 中心增强测试与启用反馈。
3. 定时任务改为任务详情布局。
4. 反馈日志独立成排查页。

## 视觉问题

当前视觉并非完全不可用，但有明显模板感：

- 白底卡片过多，页面像后台模板。
- 颜色高度依赖蓝色，层级区分不够。
- 大量圆角卡片和统计卡让页面显得“功能堆叠”。
- 中文字体和字号层级偏系统默认，缺少产品气质。
- 卡片内外边框相似，主次不清。

建议视觉方向：

- 保留浅色专业工具基调。
- 减少大卡片，增加分栏、列表、检查器、工具条。
- 主色降低饱和度，增加状态色和中性色层级。
- 数字、状态、路径使用等宽字体，但不要让路径占据主舞台。
- 图片和视频页面用深色预览区承载媒体，但工具区仍保持浅色。

## 后续实现原则

每次改一个页面前，先写该页面的一句话任务定义：

```text
这个页面帮助用户完成什么判断或动作？
```

然后再决定布局，不允许先堆功能。

每个页面至少定义：

- 主对象
- 主操作
- 次操作
- 空状态
- 失败状态
- 批量状态
- 与其他页面的流转关系

每个实现 PR 的验收方式：

- 跑真实 Electron 页面。
- 截图保存桌面宽度。
- 检查长中文、路径、按钮文案不溢出。
- 高频流程必须走完一次真实点击。

## 建议先改的具体任务

第一张开发票据建议写成：

```text
重构图片工作室验收阶段：新增大图查看器、左右切换、缩放、通过/拒绝快捷键、来源提示词面板，并将生成完成后的默认落点改为验收工作台。
```

第二张开发票据：


这三项完成后，产品的“功能及格但不好用”的问题会明显缓解，因为它们覆盖了最核心的判断流程：看图、看视频、看发布计划。
