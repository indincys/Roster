# 第一部分：产品总览与全局架构

## 一、产品概览

### 1.1 产品定位

跨平台（macOS + Windows）桌面工作台软件，面向独立或小团队的短视频带货 / 品牌运营者，聚合"任务单生成 + AI 内容生产 + 数据库管理"等多个工作流。macOS 与 Windows 两端功能完全一致，用户可在任一端完成创作、管理、任务单导出、状态查看和设置维护。**本地优先 + 云同步友好**：所有数据在本地存储，但通过 OneDrive / 坚果云等云盘工具支持多台工作电脑切换；影刀 RPA 作为外部 Windows 执行器读取任务单并回写状态。

### 1.2 v1 范围

**v1 必做**：

- 多品牌工作空间（隔离）
- 6 个数据库（视频/标签/标题/提示词/图片/文案）
- 任务单生成 + 编辑 + 导出 + RPA 状态回写
- 标题、图片、文案三个 AI 工作流
- 封面工作区（手动拉时间轴）
- Skill 系统（含 include 机制 + 版本 + 云端同步）
- Skill 市场基础版
- 定时任务系统
- 完整设置模块
- 备份与日志

**v1 不做**：

- 暗色模式
- AI 推荐封面
- 封面创作（加标题、滤镜）
- 多人协作
- 渐进式披露 Skill（让 LLM 用工具读子文件）
- 移动端
- 首次使用引导教程
- 自动汇总反面库优化 Skill（数据先收集，能力后续做）

### 1.3 核心非功能性需求

| ID             | 需求     | 标准                                      |
| -------------- | ------ | --------------------------------------- |
| NFR-GLOBAL-001 | 跨平台一致性 | macOS 与 Windows 功能 100% 对等，UI 视觉差异 < 5% |
| NFR-GLOBAL-002 | 启动速度   | 冷启动到主界面可交互 ≤ 5 秒（普通配置机）                 |
| NFR-GLOBAL-003 | 内存占用   | 空闲态 < 600MB；活跃态（含图片网格）< 1.2GB           |
| NFR-GLOBAL-004 | 列表流畅度  | 任意列表 60fps 滚动；不卡顿                       |
| NFR-GLOBAL-005 | 数据安全   | 任何写入操作必须保证原子性，崩溃后数据库不损坏                 |
| NFR-GLOBAL-006 | 离线可用   | 除云端 Skill 同步外，所有功能离线可用                  |
| NFR-GLOBAL-007 | 中文界面   | 所有 UI 文本中文优先；英文仅用于变量名、API 字段、日志         |
| NFR-GLOBAL-008 | 双端一致性  | 除系统级文件选择器、路径格式、凭证底层实现等 OS 差异外，macOS 与 Windows 不存在功能缺口 |

## 二、整体架构

### 2.1 技术栈

详见 `技术方向决定备忘录.md`。核心：

```
Electron + React + TypeScript + Tailwind CSS + shadcn/ui + Zustand
+ SQLite (better-sqlite3) + chokidar + fluent-ffmpeg + exceljs + Octokit
+ electron-builder
```

### 2.2 进程架构

```
┌────────────────────────────────────────────────┐
│  渲染进程（Renderer）                            │
│  React UI / 用户交互 / 状态管理                  │
│  ↑ ↓ IPC（类型化）                              │
├────────────────────────────────────────────────┤
│  Preload（contextBridge）                       │
│  暴露受限 API 给渲染进程                         │
├────────────────────────────────────────────────┤
│  主进程（Main）                                  │
│  · 窗口管理     · IPC server                    │
│  · 系统托盘     · 菜单                          │
│  · 文件 I/O    · 数据库（SQLite）                │
│  · API 调用    · 调度器                         │
├────────────────────────────────────────────────┤
│  Worker / 子进程                                 │
│  · ffmpeg 拉帧                                   │
│  · 大批量数据库操作                              │
│  · 大文件处理                                    │
└────────────────────────────────────────────────┘
```

### 2.3 数据持久化布局

#### 软件全局目录（用户隔离）

- **macOS**：`~/Library/Application Support/MyWorkbench/`
- **Windows**：`%APPDATA%\MyWorkbench`

```
MyWorkbench/
├── config.db                  # 全局 SQLite：API key、工作空间索引、用户偏好
├── skills/                    # 全局 Skill 池
│   ├── official/              # 官方原版（只读，云端同步）
│   ├── user/                  # 用户自建或副本
│   └── manifest.json          # 官方 Skill 版本清单（云端同步）
├── vault/                     # 可选：跨设备凭证保险箱（用户主动启用）
├── cache/                     # 缩略图、API 响应缓存
└── logs/                      # 滚动日志
```

默认全局目录是本机私有目录。若用户希望在多台工作电脑间同步全局配置，可在设置中启用“同步配置目录”，把工作空间索引、用户自建 Skill、启用配置和可选凭证保险箱放到云盘同步目录。缓存、缩略图、临时文件和普通日志默认不进入同步配置目录。

#### 工作空间目录（用户选择，建议放云盘）

```
品牌A/
├── workspace.db              # 该空间的 SQLite（视频/标题/标签/任务等）
├── workspace.json            # 元数据（路径映射、平台账号等）
├── workspace.lock            # 当前写入实例锁文件（运行时创建，退出时删除）
├── videos/                   # 视频原文件（按 SKU 子文件夹）
│   └── SKU-WB-001/
│       ├── style_a.mp4
│       └── style_b.mp4
├── covers/                   # 封面文件（镜像 videos 结构）
│   └── SKU-WB-001/
│       ├── style_a__3x4.jpg
│       └── style_a__9x16.jpg
├── images/
│   ├── main/                 # 主图素材
│   ├── detail/               # 详情页素材
│   ├── live_cover/           # 直播封面
│   └── _trash/               # 软删图（按配置定期真删）
├── tasks/
│   └── 2026-05-07/
│       ├── tasks.xlsx
│       ├── tasks.csv
│       ├── tasks.json
│       ├── preflight.json
│       └── status/           # RPA 写回的状态文件
│           ├── T20260507-001__attempt-001.json
│           └── T20260507-002__attempt-001.json
├── skills_config/            # 该空间启用了哪些 Skill 的配置
│   └── activation.json
└── _backup/                  # 自动备份（按配置保留 N 份）
```

`workspace.db` 可以随工作空间云盘目录同步，但 v1 明确不是多人协作数据库。同一时间只允许一个 App 实例以写模式打开同一工作空间。第二台电脑检测到有效 `workspace.lock` 和心跳时，应进入只读模式或提示用户等待同步完成后再打开。

### 3.3 文件类约定

#### Skill 包结构

```
<skill_name>/
├── meta.json                # Skill 元数据（类型、版本、来源、变量定义等）
├── SKILL.md                 # 主文件（必须存在，作为 LLM 入口）
├── *.md                     # 子文件（被 SKILL.md 通过 {{include: ...}} 引用）
└── examples/                # 示例（可选）
    └── *.md
```

**meta.json 字段**：

```json
{
  "name": "title_brand_warmlife",
  "display_name": "暖心生活标题 Skill",
  "type": "title",                       // 'title' | 'image_prompt' | 'script'
  "version": "1.2.0",
  "origin": "official/title_general@1.0.0",  // null 表示自建
  "created_at": 1715000000,
  "updated_at": 1715050000,
  "default_model": "claude-opus-4-7",
  "supported_models": ["claude-opus-4-7", "gpt-5", "gemini-2.5"],
  "description": "..."
}
```

#### 任务单状态文件（status/<run_key>.json）

由 RPA 写入，由软件读取：

```json
{
  "schema_version": 1,
  "task_id": "T20260507-001",
  "attempt_no": 1,
  "run_key": "T20260507-001__attempt-001",
  "status": "success",
  "executed_at": "2026-05-07T09:08:34Z",
  "platform_post_url": "https://...",
  "error_code": null,
  "error_message": null,
  "rpa_log": "可选的执行日志文本",
  "writer": "yingdao-rpa"
}
```

RPA 写状态文件时必须先写入 `<run_key>.tmp`，写完并关闭文件句柄后再原子重命名为 `<run_key>.json`。软件只读取 `.json`，忽略 `.tmp`。同一 `run_key` 的状态事件必须幂等处理；云盘重复触发文件变更时不得重复执行副作用。

#### 路径预检文件（preflight.json）

任务单导出时同时生成：

```json
{
  "schema_version": 1,
  "generated_on": "macos",
  "target_platform": "windows",
  "workspace_id": "ws_xxx",
  "task_date": "2026-05-07",
  "items": [
    {
      "task_id": "T20260507-001",
      "relative_path": "videos/SKU-WB-001/style_a.mp4",
```

# 第二部分：核心模块详细规格

## 四、工作空间系统（WS）

### 4.1 目的

每个工作空间 = 一个独立运营的品牌。空间之间数据完全隔离，包括视频、标签、标题、Skill 启用配置等。一个软件实例支持管理多个工作空间。

### 4.2 用户场景

- 用户在抖音同时运营 3 个品牌（暖心生活、精致时刻、野趣户外），每个品牌的视频、标题风格、调用 Skill 完全不同
- 切换工作空间后，整个软件感觉就像是"另一个独立运营的工具"
- 添加新品牌时不影响现有品牌的数据

### 4.3 功能需求

| ID        | 需求                                                               |
| --------- | ---------------------------------------------------------------- |
| FR-WS-001 | 用户可创建新工作空间，需指定：名称、根目录路径（Mac 与 Win 各一个）、可选模板（空 / 复制现有空间 Skill 配置） |
| FR-WS-002 | 用户可通过顶部条切换器一键切换工作空间，切换后整个主内容区数据刷新                                |
| FR-WS-003 | 用户可重命名工作空间                                                       |
| FR-WS-004 | 用户可删除工作空间，二次确认后软件不再索引该空间，但**不删除磁盘上的实际文件**                        |
| FR-WS-005 | 软件启动时自动打开"上次使用"或"标记默认"的工作空间                                      |
| FR-WS-006 | 工作空间切换时，正在运行的 AI 调用自动取消（防止把 A 空间的结果写到 B 空间）                      |
| FR-WS-007 | 每个工作空间有独立的：6 个数据库内容、Skill 启用配置、平台账号清单、生成预设、定时任务列表                |
| FR-WS-008 | 全局共享：API key、Skill 池本身（具体启用哪些是空间级配置）、软件设置                        |
| FR-WS-009 | 同一工作空间同一时间只允许一个 App 实例写入；检测到其他设备有效锁和心跳时，本机必须进入只读或提示等待              |

### 4.4 技术实现要点

- **工作空间索引**存于全局 `config.db.workspaces` 表
- **工作空间数据**存于该空间根目录下的 `workspace.db` (SQLite) + `workspace.json` (元数据)
- 切换工作空间 = 关闭当前 `workspace.db` 连接 + 打开新空间的 `workspace.db` + 触发全 UI 刷新
- 工作空间根目录路径变更时（如用户挪了文件夹），需提供"重新指向"功能而不是重新创建
- 打开工作空间写模式前，主进程创建或刷新 `workspace.lock`，内容包含 device_id、os、app_instance_id、opened_at、last_heartbeat_at
- 若发现锁文件仍在有效心跳窗口内，不得强行写入 `workspace.db`

### 4.5 关键流程

**4.5.1 创建工作空间**

1. 用户点"新建工作空间" → 弹层
2. 输入名称、Mac 路径、Win 路径（可空，Win 路径以后再补）
3. 软件检测路径有效性（路径存在、可写、容量充足）
4. 软件创建目录结构（按 §2.3）
5. 初始化 `workspace.db`（运行 schema SQL）
6. 写入全局 `workspaces` 表
7. 切换到新空间

**4.5.2 切换工作空间**

1. 用户在顶部下拉选择目标空间
2. 软件检查当前空间是否有未保存编辑 → 若有，提示保存
3. 取消所有正在执行的 AI 调用
4. 关闭当前 SQLite 连接
5. 打开目标空间的 `workspace.db`
6. 重新拉取 UI 数据
7. 更新顶部条显示
8. 写入 `last_opened_at`

### 4.6 验收标准

| ID        | 类型 | 标准                                                                                        |
| --------- | -- | ----------------------------------------------------------------------------------------- |
| AC-WS-001 | 功能 | 创建工作空间后，磁盘上能看到完整目录结构，包括 videos / covers / images / tasks / skills\_config / \_backup 等子目录 |
| AC-WS-002 | 功能 | 创建空间后立即可在切换器中看到，且能成功切换                                                                    |
| AC-WS-003 | 功能 | 切换工作空间后，所有数据库列表、Skill 启用、平台账号都变成新空间的内容                                                    |

## 五、路径管理与跨平台（PATH）

### 5.1 目的

软件在 macOS 与 Windows 两端功能完全一致，用户可以在任一端创作、管理和导出任务单。两端通过云盘（OneDrive / 坚果云）共享同一个工作空间文件夹。macOS 和 Windows 上的路径前缀格式不同，必须有一套机制保证：**软件内部统一存储相对路径**，**UI 按当前设备展示本机路径**，**导出给 RPA 时按目标执行平台拼接绝对路径**。v1 的 RPA 目标执行平台默认为 Windows 影刀。

### 5.2 功能需求

| ID          | 需求                                                                    |
| ----------- | --------------------------------------------------------------------- |
| FR-PATH-001 | 工作空间元数据中保存 Mac 根路径与 Win 根路径两个独立配置                                     |
| FR-PATH-002 | 数据库中所有路径字段只存相对工作空间根目录的相对路径，不存绝对路径                                     |
| FR-PATH-003 | UI 展示文件路径时，按当前操作系统拼接为绝对路径                                             |
| FR-PATH-004 | 任务单导出时，RPA 路径字段按目标执行平台拼接；v1 默认目标为 Windows，因此默认写 Win 绝对路径                  |
| FR-PATH-005 | 任务单导出前自动校验：所有引用文件在当前设备存在、可读、可解析；缺失或疑似未同步文件高亮警告，让用户决定是否继续             |
| FR-PATH-006 | 软件检测当前工作空间根目录是否在云盘同步路径下（识别 OneDrive、坚果云、Dropbox、Google Drive），不在则提示用户 |
| FR-PATH-007 | 工作空间根路径变更后（用户挪了云盘位置），提供"重新指向"功能，不破坏数据库内容                              |
| FR-PATH-008 | 每次导出同时生成 `preflight.json`，供 Windows 端软件或 RPA 执行前做目标端路径二次校验                     |
| FR-PATH-009 | UI 必须同时能展示“当前设备路径”和“RPA 执行路径”，避免用户混淆本机可读与 RPA 可读                          |

### 5.3 技术实现要点

- **路径分隔符**：相对路径统一用正斜杠 `/` 存储，跨平台展示时再转换
- **路径拼接**：使用 Node.js `path.posix.join` 处理相对路径，`path.join`（系统 native）处理最终绝对路径
- **云盘检测**：路径关键字只能作为弱提示；检测结果用于 UI 提醒，不作为唯一可靠判断
- **路径校验**：导出前执行组合校验：`exists`、`stat`、打开文件句柄、读取少量字节、视频 `ffprobe`、图片尺寸读取。不能仅靠文件 size 判断占位符
- **目标端预检**：macOS 端无法证明 Windows RPA 机器已经同步完成，因此导出时必须生成 `preflight.json`。Windows 端软件或 RPA 执行前读取该文件并在 Windows 本机验证每个目标路径
- **原子写入**：任务单、`preflight.json` 和状态文件均采用先写临时文件再重命名的方式，避免云盘同步到半成品

### 5.4 关键流程

**5.4.1 路径校验流程（导出前）**

```
1. 收集任务单中所有视频路径 + 封面路径
2. 对每条相对路径：
   a. 拼接为本地绝对路径（按当前 OS）
   b. 检查 exists + stat
   c. 打开文件并读取少量字节，确认不是不可读占位符
   d. 视频用 ffprobe 读取基础元数据；封面用图片解码读取尺寸
   e. 记录文件大小、mtime、本机校验结果
3. 生成目标执行平台路径。v1 默认目标为 Windows，使用 `win_root_path` + 相对路径，并统一为反斜杠
4. 写出 `preflight.json`，包含所有相对路径、目标路径、文件大小、mtime、本机校验结果和生成设备信息
5. 若有任何文件缺失、不可读或解析失败，弹层显示问题列表：
   - 文件名 / 当前设备路径 / RPA 执行路径 / 状态（缺失 / 不可读 / 解析失败 / 目标端待校验）
   - 让用户选择：[继续导出但标记风险] / [取消] / [打开同步目录]
6. 若导出设备是 Windows 且目标执行平台也是 Windows，软件可完成目标路径本机二次校验；若导出设备是 macOS，则目标端二次校验必须在 Windows 端完成
```

### 5.5 验收标准

| ID          | 类型 | 标准                                             |
| ----------- | -- | ---------------------------------------------- |
| AC-PATH-001 | 功能 | 数据库中所有路径字段都是相对路径（不含 `/Users/` 或 `D:` 这种绝对前缀）   |
| AC-PATH-002 | 功能 | 在 Mac 上看到的视频文件路径前缀是 Mac 根路径；在 Win 上是 Win 根路径   |
| AC-PATH-003 | 功能 | v1 默认导出的任务单 Excel/CSV/JSON 中 RPA 路径字段都是 Win 绝对路径，反斜杠分隔 |
| AC-PATH-004 | 功能 | 导出前路径校验能正确识别本地不存在、不可读、媒体解析失败或疑似未同步的文件并警告       |
| AC-PATH-005 | 功能 | 软件检测到工作空间根目录不在云盘同步路径下时，给出黄色提示 banner           |
| AC-PATH-006 | 功能 | 工作空间根路径修改后，原有的相对路径数据仍可用（拼接新前缀仍能找到文件）           |
| AC-PATH-007 | 功能 | 每次导出生成 `preflight.json`，Windows 端可读取并校验所有 RPA 目标路径        |
| AC-PATH-008 | 功能 | UI 能清楚区分当前设备路径与 RPA 执行路径                              |
| AC-PATH-201 | 边界 | 用户输入的根路径包含特殊字符（中文、空格、emoji）时，软件正常处理            |
| AC-PATH-202 | 边界 | Win 路径输入时支持 `D:\xxx` 与 `D:/xxx` 两种格式，存储时归一化    |

### 7.3 关键算法

#### 7.3.1 视频筛选算法

**热门 SKU 倾向**：

```sql
SELECT * FROM videos 
WHERE status != 'archived'
  AND sku_code IN (SELECT sku_code FROM (
    SELECT sku_code, COUNT(*) as use_count 
    FROM tasks 
    WHERE status='success' AND task_date > date('now', '-30 days')
    GROUP BY sku_code 
    ORDER BY use_count DESC 
    LIMIT 20
  ))
ORDER BY RANDOM()
LIMIT N
```

**低发倾向**：

```sql
SELECT v.* FROM videos v
LEFT JOIN tasks t ON t.video_id = v.id AND t.status = 'success'
WHERE v.status != 'archived'
GROUP BY v.id
ORDER BY COUNT(t.id) ASC, v.added_at DESC
LIMIT N
```

**近期热点**：基于"近 7 天发布平均成绩"排序（v1 暂用"近 7 天发布次数最少 + 最新入库"作为简化算法，"成绩"数据 v2 后接入平台 API 后实现）。

**自定义**：用户在 UI 选 SKU、款式、入库日期等具体过滤条件。

#### 7.3.2 时间分配算法

```python
def assign_times(anchors: list[str], jitter_min: int, count: int) -> list[str]:
    """
    anchors: 时间锚点 ['09:00', '12:00', '15:00', '18:00', '21:00']
    jitter_min: 抖动分钟数（如 15）
    count: 需要的时间数
    
    按 round-robin 分配任务数量。count 不能被 anchors 数整除时，余数分配给前若干锚点，不得丢任务。
    每个时间 = 锚点 ± random(0, jitter_min*60) 秒。
    """
    if count <= 0:
        return []
    if not anchors:
        raise ValueError("anchors must not be empty")

    times = []
    base = count // len(anchors)
    remainder = count % len(anchors)

    for index, anchor in enumerate(anchors):
        quota = base + (1 if index < remainder else 0)
        for _ in range(quota):
            jitter_seconds = random.randint(-jitter_min*60, jitter_min*60)
            t = parse_time(anchor) + timedelta(seconds=jitter_seconds)
            times.append(t.strftime('%H:%M:%S'))

    times.sort()
    return times
```

### 9.4 Skill include 解析器规范

```typescript
interface IncludeResult {
  text: string;            // 展开后的完整文本
  errors: string[];        // 解析过程中的错误（如循环引用、文件缺失）
}

function expandSkill(skillPath: string): IncludeResult {
  const visited = new Set<string>();
  const errors: string[] = [];
  const rootReal = fs.realpathSync(skillPath);
  const maxBytes = 256 * 1024;

  function resolveInsideSkill(fromDir: string, includeName: string): string | null {
    if (path.isAbsolute(includeName)) return null;
    if (includeName.split(/[\\/]/).some(part => part === '..' || part.startsWith('.'))) return null;
    if (path.extname(includeName) !== '.md') return null;

    const candidate = path.resolve(fromDir, includeName);
    if (!fs.existsSync(candidate)) return candidate;

    const real = fs.realpathSync(candidate);
    if (!real.startsWith(rootReal + path.sep)) return null;
    return real;
  }
  
  function expand(filePath: string, depth: number): string {
    if (depth > 5) {
      errors.push(`Max depth exceeded at ${filePath}`);
      return '';
    }
    if (visited.has(filePath)) {
      errors.push(`Circular reference: ${filePath}`);
      return '';
    }
    visited.add(filePath);
    
    if (!fs.existsSync(filePath)) {
      errors.push(`File not found: ${filePath}`);
      return '';
    }

    const stat = fs.statSync(filePath);
    if (!stat.isFile() || stat.size > maxBytes) {
      errors.push(`Invalid include file: ${filePath}`);
      return '';
    }
    
    let content = fs.readFileSync(filePath, 'utf-8');
    content = content.replace(/\{\{include:\s*([^}]+)\}\}/g, (_, fname) => {
      const subPath = resolveInsideSkill(path.dirname(filePath), fname.trim());
      if (!subPath) {
        errors.push(`Include outside skill sandbox or invalid file type: ${fname}`);
        return '';
      }
      return expand(subPath, depth + 1);
    });
    
    visited.delete(filePath);  // 允许同级别非循环复用
    return content;
  }
  
  const entry = path.join(rootReal, 'SKILL.md');
  const text = expand(entry, 0);
  return { text, errors };
}
```

# 第三部分：交付与质量

## 十五、开发分期与里程碑

建议分 7 个里程碑，每个里程碑产出可独立验收。

### M1：骨架与基础设施（约 2 周）

**目标**：搭好 Electron + React + SQLite 全链路，跑通最小 demo。

**交付**：

- 项目初始化（monorepo 结构）
- IPC 类型化通信
- 全局 SQLite 初始化
- 工作空间创建 / 切换骨架
- 顶部条 + 左侧栏 + 首页（数据为空也能打开）
- 路径映射机制
- 设置页基础版（API key 录入）

**验收**：能创建工作空间、切换、API key 加密存储、软件启动 < 5 秒。

### M2：数据库与视频库（约 2 周）

**目标**：6 个数据库的列表骨架 + 视频库完整功能。

**交付**：

- 通用数据库列表组件
- 视频库扫描 + 缩略图 + CRUD
- 标签库手动维护
- 标题库 / 提示词库 / 图片库 / 文案库 列表（先空着，后续模块填充）

**验收**：1 万条视频列表流畅滚动；扫描功能正确。

### M3：任务单核心（约 3 周）

**目标**：任务单生成 + 编辑 + 导出 + 状态回写完整跑通。

**交付**：

- 任务单生成算法（4 种视频筛选 + 3 种标题策略 + 时间分配）
- 任务单 UI 表格 + 行内编辑
- Excel/CSV/JSON 导出
- 路径校验
- 状态文件扫描 + 回写
- 数据库回写（视频 used、标题 use\_count）
- 历史任务单查看

**验收**：能完整跑通"生成 → 导出 → RPA 模拟写状态 → UI 状态更新"全链路。

### M4：Skill 系统与标题工作区（约 3 周）

**目标**：Skill 引擎 + 标题工作区完整可用。

**交付**：

- Skill 文件夹结构 + meta.json + include 解析器
- Skill 中心 UI（编辑器、版本、测试、复原）
- Skill 市场基础（GitHub 拉 manifest + 安装）
- LLM Provider 抽象层 + Anthropic + OpenAI 实现
- 标题工作区（多模型对比 + 流式 + 入库）

**验收**：能从市场安装一个标题 Skill，调用生成 20 条标题入库，被任务单使用。

### M5：图片工作室与封面工作区（约 3 周）

**目标**：图片完整工作流 + 封面手动管理。

**交付**：

- 图片工作室 3-tab
- 提示词库
- 图片 Provider 抽象层 + 至少 GPT Image 2 实现
- 软删 + 反面库 + 入库率
- 封面工作区（ffmpeg 拉帧 + 时间轴 + 比例裁剪）

**验收**：能从一条种子描述生成 20 条提示词，跑出 60 张图，挑选入库；能批量给 30 个视频选封面。

### M6：定时任务、文案、其他完善（约 2 周）

**交付**：

- 定时任务系统 + 总览
- 文案工作区（基础版）
- 完整设置模块
- 备份与恢复
- 日志系统

**验收**：定时生成任务单功能正常；备份与恢复可用。

### M7：稳定性、性能、打包分发（约 2 周）

**交付**：

- 性能优化（虚拟列表、缩略图缓存、IPC 优化）
- 跨平台测试（Mac + Win 全功能验证）
- 双平台签名打包（electron-builder）
- 自动更新机制
- 完整日志清理 / 备份等后台任务
- 文档（用户手册）

**验收**：所有非功能性需求达标；可对外分发安装包。

**总计**：约 17 周（4 个月，单工程师全职估算；多工程师可并行缩短）
